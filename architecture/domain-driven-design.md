# 도메인 주도 설계(Domain-Driven Design)

---

### 도메인 주도 설계(DDD)가 무엇인가?

DDD는 도메인(Domain)을 바탕으로 서비스를 설계하는 방식이다.

비즈니스의 구조가 그대로 코드의 구조에 반영되기 때문에 소통과 판단이 용이해진다.



### DDD의 Tradeoff

- 장점
    - 서비스의 확장성과 유지보수성이 올라간다.
    - 비즈니스 로직의 무결성을 지킬 수 있다.
- 단점
    - api나 이벤트 등의 호출 비용이 발생한다.
    - 여러 도메인의 정보가 필요해지는 순간 조회 비용이 증가한다.



### 바운디드 컨텍스트(Bounded Context)

바운디드 컨텍스트는 도메인의 각 영역 별 경계선이다.

업무 규칙이 섞이지 않도록 하는 울타리라고 볼 수 있다.

서로 소통은 해야 하지만 직접 연관되지 않은 영역들끼리도 종속성이 생기면 코드의 응집성이 과도하게 높아진다.

따라서 DDD에서는 서로 분리되는 영역간에는 종속성을 제거하고, facade나 api, 이벤트 등을 통해 소통하는 형식을 취하게 한다.

### 애그리거트(Aggregate)와 애그리거트 루트(Aggregate root)

바운디드 컨텍스트 내에서 일관성을 가져야 하는 서로 연관된 모델의 모음이 애그리거트(Aggregate)다.

그리고 해당 애그리거트와 소통하기 위한 진입점을 애그리거트 루트(Aggregate root)라고 한다.




### 필수 원칙

```
1. Repository는 루트만
2. 불변식을 루트가 보장
3. 다른 애그리거트는 id 참조
```

다만 원칙에 얽매여 설계 목적과 멀어지는 일이 없도록 하자. 특히 1번 원칙은 서비스에 따라서는 어기는 일이 발생하곤 한다.



### DDD의 5대 요소

1. 엔티티(Entity) - 고유 ID를 가진 핵심 객체
    - 정적 팩토리 패턴을 활용하는 것을 추천
2. 밸류 오브젝트(Value Object) - 값 자체가 의미인 객체
    - 값을 저장할 때 규칙을 적용하기 용이함
    - 객체 생성도 결국 비용이다
3. 애그리거트(Aggregate) - 연관 객체를 하나로 묶는 단위
    - 외부에서는 애그리거트 루트를 통해서만 내부 객체에 접근한다
    - 다른 애그리거트를 참조할 때는 객체 직접 참조 대신 ID로 참조한다
    - 하나의 트랜잭션에서는 하나의 애그리거트만 수정한다
4. 리포지토리(Repository) - 애그리거트 저장/조회
    - 리포지토리는 애그리거트 루트 단위로 생성
    - VO는 리포지토리 불필요
    - 도메인과 인프라를 분리(DIP 원칙)
        - 이로 인해 도메인은 DB 기술 등과 무관
    - JPA 더티 체킹 활용
5. 도메인 서비스(Domain Service) - 특정 엔티티에 속하지 않는 로직
    - 특정 엔티티에 속하면 해당 엔티티 메서드에
    - 기술적인 관심사에 속한다면 Application Service에



### DDD의 4계증

- 표현(Presentation) : 외부와의 통신을 담당
- 응용(Application) : 서비스 흐름 제어 담당
- 도메인(Domain) : 비즈니스 로직과 규칙 잠당
- 인프라(Infrastructure) : 기술 담당



### 도메인(Domain) 계층

**"우리 서비스가 무엇이고, 무엇이 허용되고 금지되는가"**를 표현하는 계층.

비즈니스 로직과 불변식을 이 곳에서 정의하고 활용한다.

Entity에는 데이터와 Getter, Setter만 있고 규칙과 비즈니스 로직은 외부에 있는 빈약한 도메인(Anemic Domain)이 되지 않도록 경계할 필요가 있다.



**포함되는 개념**
- Entity
- VO(Value Object)
- Aggregate
- Invariant(불변식) : 어떤 순간에도 반드시 참이어야 하는 규칙

### 응용(Application) 계층

**"누가 무엇을 어떤 순서로 할 것인가?"와 "어디부터 어디까지 일괄적으로 처리되어야 하는가?"**를 제어하는 계층.

UseCase의 워크플로우와 트랜잭션을 이곳에서 정의하고 제어한다.

**응용 서비스 vs 도메인 서비스**
||응용 서비스|도메인 서비스|
|--|----|----|
|한마디로|흐름을 조율|규칙을 구현|
|언제 쓰나|UseCase 하나의 순서, 트랜잭션 관리|규칙이 한 객체에 담기 애매할 때(여러 객체에 걸칠 때)|
|트랜잭션|책임진다(@Transactional)|모른다|
|예시|"주문하기" 전체 진행|"환율 적용 송금액 계산"처럼 게좌+환율 여러 개가 얽힌 규칙|

### 표현(Presentation) 계층

**"외부와 어떻게 소통할 것인지"**를 정의하는 계층.

외부 통신의 규격을 이곳에서 정의하고 통신을 제어한다.

***절대로 내부 Entity를 외부로 직접 노출하지 말것!!!***

DTO를 사용하여 내부 엔티티를 그대로 내보내지 않고, 검증 책임도 명확히 구분한다.

### 인프라(Infrastructure) 계층

**기술적인 부분을 실제로 구현하는** 계층.

DB 등등의 외부 기술과의 연동을 이곳에서 처리한다.

**DIP를 적용하는 이유**

원칙상 윗 계층은 아래 계층을 참조한다. 하지만 그렇게 하면 도메인이 기술에 묶여버리는 사태가 발생한다.

구체적으로는 외부 기술이 변경되면 내부 로직도 같이 변경해야 하는 상황이 발생한다.

이를 분리하기 위해 DIP를 적용한다.
