# Spring Batch

### Spring Batch를 이용하는 이유



정산 도메인에서는 반드시 지켜져야 할 것들이 3가지 있다.

1. 절대로 틀리면 안된다(무결성).
2. 같은 행동을 두 번 하면 안된다(멱등성).
3. 제 시간에 끝내야 한다(성능).

금융에서는 0.1원 단위도 커다란 변동으로 다가온다. BigDecimal을 사용하는 것도 그 이유이다(float/double은 원리 상 소수점에 오류가 발생할 수 있다).

특히 정산은 금액이 매우 작은 금액이라도 차이가 발생하는 순간, 그 원인을 찾기 위해 비상이 걸린다. 코딩이슈인지, 아니면 자금의 운용에 부정이나 실수가 있었는지를 알아야 하니.

당연하지만 저건 개발자의 영역이 아닌 재무팀의 영역이다. 개발자의 실수가 재무팀을 굴리는 거다.

'단순 실수로 발생하여 금방 복구한 사고'일때의 낙관적 시나리오가 이 정도고, 심각해지면 대형 사건으로 발전할수도 있다.

이런 일이 벌어지지 않도록 저 3가지 원칙을 철저히 지킬 방법을 찾아내야 하는 것이다.

Spring에서는 그것이 Batch가 된다.

### Spring Batch의 특성

Spring Batch는 하나의 Job를 여러개의 Step으로 나누어 진행한다.

이렇게 하나의 작업을 쪼개면 작업이 대량으로 들어올 때 해당 작업을 나누어서 처리할 수 있게 된다.

그리고 건수 별로 작업 커밋을 하기 때문에 소모 메모리도 일정하게 유지할 수 있다.

- Reader : 데이터를 한 페이지씩 읽어옴
- Processor : 읽은 데이터를 가공, 변환함
- Writer : 가공된 데이터를 한 묶음(Chunk)씩 묶어 처리함

**Reader가 데이터를 읽는 방식**
|방식|어떻게 동작하나|특징|
|--|-----|-----|
|Paging|LIMIT/OFFSET으로 페이지를 끊어서 여러 번 쿼리|페이지당 메모리만 쓰니 안전. JPA랑 잘 맞음. 뒤로 갈수록 OFFSET이 무거워질 수 있음|
|Cursor|DB 커서로 연결을 열어둔 채 한 건씩 흘려보냄|OFFSET 부담은 없지만 커넥션을 길게 붙잡음|

### 멱등성(Idempotency) 유지

Batch도 언젠가는 반드시 죽는다. 영원히 문제없는 기술은 없으며, 규모가 커지고 구조가 복잡해지면 더더욱 그렇다.

그럼에도 멱등성은 유지되어야 한다. 서비스가 한번 에러를 일으켰다고 결제를 두 번 하는 일이 있어서는 안된다.

보통은 하단의 세가지 구조를 이용해서 방어를 한다.

1. UNIQUE 제약 - id에 유일 제약을 걸어 db가 동일한 정보를 다시 처리하지 않도록 한다.
2. 상태값 확인 - 상태값이 DONE으로 처리된 데이터를 Reader가 거부하거나 Processor가 건너뛴다.
3. Restartability - Spring Batch가 어디까지 처리했는지 기록해 재실행 시 해당 이후부터 진행한다.

### Restartability

Restartability는 **식별 → 기억 → 재개**의 과정을 거치는 식으로 구현된다.

**식별**

JobInstance를 Job 이름과 JobParameters의 조합으로 식별한다. 이때, 현재 작업과 동일한 식별자를 가졌지만 중간에 실패하여 중단된 인스턴스가 발견되면 새 인스턴스를 생성하지 않고 해당 인스턴스를 재시작한다.

**기억**

Chunk의 commit이 성공하면, Batch는 현재까지의 진행상태를 ExecutionContext에 담아 DB에 같이 저장한다.
